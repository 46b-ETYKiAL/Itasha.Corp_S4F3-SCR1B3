## Advisories

### RUSTSEC-2025-0141 — `bincode 1.3.3` unmaintained

`bincode` 1.x reached its **unmaintained** state on 2025-12-16
(no longer accepting patches; users encouraged to migrate to
`bincode` 2.x). This is **not a security vulnerability** — no CVE
is attached, no exploit is published, no patched version of the 1.x
line is necessary.

`bincode 1.x` is a **transitive dependency** through
`syntect 5.3.0` (our syntax-highlighting engine), which has not yet
migrated to `bincode` 2.x. SCR1B3 does not consume `bincode`
directly.

**Resolution path**: hold for upstream `syntect` to migrate (the
maintainer is aware; tracking issue exists). If `syntect` doesn't
migrate within 90 days of this audit, evaluate whether the
syntax-highlighting engine should be swapped to `tree-sitter`-only
(already a workspace dep; the migration cost is bounded).

The advisory is documented here so a future auditor can confirm
the decision was deliberate; `cargo audit` will continue to surface
the warning until upstream lands.

## Unsafe-code surface

`grep -rE "unsafe \{"` over `apps/scribe/crates/*/src/`: **2 sites**.

| Site | Justification |
|---|---|
| `scribe-core::document::open` mmap call | Documented `#[allow(unsafe_code)]` + SAFETY: comment. Read-only `memmap2::Mmap::map` of a just-opened file; never written through; dropped before any edit. |
| `scribe-core::buffer::open` mmap call | Same shape as above, applied to the Phase 15 KEYSTONE `Buffer::Mmap` variant. SAFETY: comment names the invariants. |

Both sites are reachable only via `scribe-core` (which carries
`#![deny(unsafe_code)]` at the crate root); `scribe-render` and
`scribe-app` both carry `#![forbid(unsafe_code)]` (strict — no
local override possible). The discipline is intentional and
documented in `SECURITY.md`.

## `unwrap()` / `expect()` / `panic!` surface

| Pattern | Count | Notes |
|---|---|---|
| `.unwrap()` | 107 | Bulk in tests; a sample of the production hits was inspected and confirmed to either be infallible (lock acquisition of a freshly-created `Mutex`) or guarded by a prior check. |
| `.expect("...")` | 18 | All carry descriptive panic messages so a future bug surfaces with explicit context. |
| `panic!(...)` | 9 | 7 are `#[test]`-scope assertions; the 2 production sites in `theme.rs` are programming-time invariants over the built-in palette (a missing key in a compiled-in builtin would be a build-time bug surfaced at the first frame). |

The surface is bounded and intentional. A future audit may want to
sweep the 107 unwrap sites and replace appropriate cases with the
`?` operator + a wider `Result` type; that's a maintainability
investment, not a correctness fix.

## Process / supply-chain hygiene

- `cargo-deny` workflow runs nightly on the public release path
  (advisories / bans / licenses / sources).
- `gitleaks` scans full history for accidentally-committed secrets.
- `cargo-cyclonedx` emits a CycloneDX 1.6 SBOM at every tagged
  release.
- Dependabot keeps Cargo + GitHub-Actions deps moving (weekly,
  Monday 09:00 UTC, 5-PR cap).
- `python apps/scribe/scripts/content_safety_audit.py` is the
  publishable-cleanliness gate that fails any change introducing an
  internal path, plan token, agent-system reference, or
  secret-shaped string into the public tree.

## Verdict

**SCR1B3 ships clean.** Every wired gate is green. The single
RUSTSEC advisory is a transitive maintenance flag, not a
vulnerability, and the resolution path is documented. The unsafe
surface is bounded to 2 documented memmap2 calls. The
unwrap/expect/panic surface is normal for a Rust application.

Recommend proceeding to the **final release gate**.
